# 一、Product Spec（产品规格）

Product Spec 定义系统的功能边界、业务规则与协商语义。面向角色：产品经理、业务分析师、测试工程师。

## 1.1 协商协议（Negotiation Protocol）

定义 Preference Agent 与 Mediator Agent 之间的消息类型与语义：

|**消息类型**|**发送方 -> 接收方**|**内容说明**|
|-|-|-|
|PREFERENCE|User -> Pref Agent|用户的自然语言偏好描述|
|DECLARE|Pref Agent -> Mediator|结构化偏好声明（类型/时长/红线）|
|PROPOSAL|Mediator -> Pref Agent|候选影片提案及推荐理由|
|VOTE|Pref Agent -> Mediator|评分（1-10）+ 赞成/否决/弃权|
|FEEDBACK|Pref Agent -> Mediator|定性反馈（满意/不满意的原因）|
|FINAL|Mediator -> All|最终裁决 + 群体满意度报告|

## 1.2 妥协规则（Compromise Rules）

Mediator Agent 在检测到冲突时，按以下优先级应用妥协规则：

|**规则 ID**|**规则名**|**触发条件**|**执行动作**|
|-|-|-|-|
|R1|类型交叉|两方偏好类型不同但可交叉|推荐交叉类型影片（如科幻+动作）|
|R2|时长中位数|各方可接受时长范围重叠|取中位数，超时长影片降级排序|
|R3|历史加权|VectorDB 中有相似群体记录|低评分类型自动降权，高评分类型升权|
|R4|轮流优先|多方偏好完全不可交叉|本次 Alice 优先，下次 Bob（状态持久化）|
|R5|安全牌兜底|所有规则均无法生成可行方案|推荐豆瓣 8.0+ 喜剧片（群体接受度最高）|

## 1.3 否决条件（Veto Conditions）

以下任一条件触发时，当前提案被否决，协商回到 resolve 状态：

|**否决 ID**|**条件名**|**判定标准**|
|-|-|-|
|V1|硬约束冲突|提案触及任何用户的明确红线（如恐怖片、超过最大时长）|
|V2|评分低于阈值|任何 Preference Agent 对提案评分低于 3/10|
|V3|多数否决|超过 50% 的成员明确否决|
|V4|历史重复|该影片在最近 3 个月内已被群体观看过|

否决条件确保少数派利益不被多数决碾压。每次否决触发后，Mediator 记录否决原因并调整下一轮提案策略。
